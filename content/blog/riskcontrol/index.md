## 业务风控题

**1）如何定义“营销资金安全”的核心目标？你会看哪些指标？**

**答题要点：**

* 目标不是“零风险”，而是**在可控风险下最大化业务收益**（GMV/ROI/投放规模）。
* 指标分三层：

  * **损失类**：疑似/确认作弊金额、赔付金额、异常退款金额、补贴/返利被套利金额
  * **业务影响类**：误杀率（拦截正常商家/达人/用户）、放行风险率、转化率/GMV/ROI 变化
  * **效率类**：命中率、处置时延（发现→拦截→复盘）、人工审核量/通过率
  
**一句话总结**：用“损失-收益-体验-效率”四象限建立北极星 + 护栏指标。

**2）发现某类产品在推广投放后的退款率、补贴消耗突然飙升，你怎么排查？**

**思路框架（面试很加分）：**

1. **确认口径**：退款率分母是下单还是支付？时间窗是下单日还是退款日？
2. **分层定位**：按类目→商家→计划→素材→人群包→地域/设备→渠道/联盟分解贡献（Top N）
3. **对照组**：和历史同周期、同类目、同活动（大促/补贴政策变化）对比
4. **判断性质**：

   * 活动/策略变更导致的“真波动”
   * 交易链路异常（支付、券核销）
   * 风险行为集中爆发（异常账号/订单模式）
5. **处置方案**：先止血（限额/灰度/提高门槛）→再定性（规则/模型/审核）→复盘沉淀指标

**3）“风险与业务收益平衡”怎么做？给一个可落地的方法**

**可落地做法：**

* 把策略看成一个二分类拦截器：阈值越严 → 损失下降但误杀上升、GMV 下降。
* 选择阈值的方法：

  * 设定**成本矩阵**：放过 1 单风险损失 = L；误杀 1 单正常损失 = B（GMV/ROI/商家体验）
  * 目标：最小化 `L*FN + B*FP` 或最大化 `收益 - 风险损失 - 误杀损失`
* 实操：**灰度+A/B**，在不同阈值下对比“拦截金额、误杀率、GMV/ROI”。


**4）如果你要做“黄牛/羊毛党”治理，策略体系怎么搭？**

**建议答案结构：**

* **分阶段**：准入（注册/实名/设备）→行为（浏览/加购/下单/支付/退款）→履约（签收/退货）→资金（补贴/返利/提现）
* **分手段**：规则（强约束）+ 模型（软识别）+ 机制（成本提高）+ 联防（黑名单/风控中台/商家侧）
* **分人群**：新客/老客、达人/商家、计划/素材，分别设置不同阈值和运营策略（例如新客更谨慎但需要留出增长空间）


**5）你会如何写一份“防控方案需求文档”（PRD/BRD）？**

**高频考点**：跨团队协作与项目推进能力
**模板回答：**

* 背景与问题：风险现象、规模、影响范围
* 目标与指标：损失下降 X%、误杀 < Y%、审核量 < Z
* 方案：策略链路（命中→处置→申诉→复核）、规则/模型/名单、阈值与灰度
* 数据需求：字段、埋点、口径、看板
* 风险与兜底：误杀申诉、回滚机制、紧急开关
* 排期与分工：研发/数仓/算法/运营/法务（如需）



**6）“实时监控异常波动”你会怎么做报警体系？**

**要点：**

* 指标分层：全站 → 类目/业务线 → 商家/计划 → 关键链路
* 报警规则：同比/环比、分位数阈值、EWMA/异常检测、突变点
* 报警质量：降噪（合并告警、抑制、冷却时间）、可解释（贡献度 Top N）
* 处置闭环：告警→工单→结论（真异常/误报）→规则迭代

**7）如果你是灰产或者刷单的商家，你会怎么反对抗风控系统”**

**1）对抗的“方向”通常是什么（高层次抽象）**

> 攻击者通常会尝试：

* **伪装成正常行为**：让行为分布更像真实用户/真实商家
* **分散与低频**：把风险信号稀释到多账号、多计划、多链路
* **团伙协同**：利用账号-设备-地址-支付工具的关联做“分工”
* **利用规则/政策缝隙**：比如活动规则、补贴门槛、结算/退款时间窗
* **试探系统边界**：小流量测试阈值，找到“刚好不触发”的区域

**2）我们怎么防（分层体系，而不是单点规则）**

> 我会把防护做成“多层+多信号”的体系：

* **准入与门槛层**：资质/行为门槛、分层额度、冷启动更谨慎
* **实时拦截层**：规则底线 + 风险评分排序，命中后分级处置（限额/二次校验/延迟结算等）
* **关系与团伙层**：做实体关联图谱（账号-设备-地址-支付等），抓“组织化风险”
* **事后追溯层**：结算/退款后验，黑样回流，形成样本闭环
* **动态对抗层**：灰度发布、阈值动态化、告警驱动迭代，避免策略“一刀切、被摸透”

**3）怎么证明有效（量化验收）**

> 用 A/B 或灰度看三类指标：

* **挽损**：异常补贴/退款/作弊金额下降
* **体验护栏**：误杀率、申诉通过率、GMV/ROI 影响
* **运营效率**：人工审核量、处置时延、告警质量

本质上风控**不是追求“永远抓完”**，而是通过对抗思维把**灰产成本抬高**、把**发现速度变快**，同时把对正常业务的影响控制在可接受范围。

## 建模题

**1）规则和模型怎么选？什么时候“规则优先”，什么时候“模型优先”？**

**推荐回答：**

* 规则优先：风险形态明确、强约束、需要可解释/可审计、冷启动
* 模型优先：风险形态多变、规则难覆盖、需要更高召回/自动化
* 最佳实践：**规则做底线（硬拦截/限额）+ 模型做排序（风险评分）+ 人审做兜底**



**2）如果让你做一个“风险评分卡”，你会用哪些特征？**

**按实体分组说更像专业风控：**

* 账号/主体：注册时长、实名状态、历史违规、关联账号数
* 设备/环境：设备指纹稳定性、IP/地域漂移、代理/异常网络信号
* 行为序列：短时间高频操作、路径异常（跳步）、夜间集中
* 交易/资金：客单价分布异常、退款率、补贴占比、同地址/同设备多单
* 关系网络：账号-设备-地址-支付工具的关联图谱（团伙识别）

（面试时强调：**特征要可落库、可实时、可解释**）



**3）怎么评估一个风控策略/模型？只说AUC够吗？**

**答：不够。**

* 风控更关心：Precision/Recall、KS、PR-AUC、拦截金额/挽损、误杀率、申诉通过率、人工审核节省
* 线上还要看：GMV/ROI 影响、用户/商家体验指标（下单转化、支付成功率）
* 强调：**离线指标 + 线上业务指标**一起验收
GMV 和 ROI 在电商投放/风控里几乎是天天挂在嘴边的两个核心指标，我把**定义、公式、口径坑点、以及风控场景怎么用**一次讲清楚。

**4）GMV/ROI 是什么？**

**GMV（Gross Merchandise Value）= 成交总额 / 交易总额**
通常指一段时间内产生的订单金额总和（偏“规模”）。

常见口径

* **下单GMV**：按“下单时间”统计订单金额（含未支付/后续退款的单，视平台口径）
* **支付GMV**：按“支付时间”统计实际支付金额
* **确认收货GMV**：按“确认收货/结算”口径，更贴近最终收入
* **净GMV（Net GMV）**：扣除退款/取消后的 GMV

> 面试时可以主动说一句：**“我会先对齐GMV口径：下单/支付/净额/结算”**，很加分。

**ROI（Return on Investment）= 投入产出比**

在投放里一般理解为：**花出去的广告费/补贴，带回来的成交金额**（或收益）。

**最常见（投放）ROI 口径**

* **ROI = GMV / 广告消耗（Spend）**

  * 例：Spend=10万，GMV=50万 → ROI=5

**也会出现的变体（更接近利润）**

* **利润ROI = 利润 / 投入**
* **边际ROI（mROI）**：增加 1 块钱投放，GMV/利润多了多少（用于调控）

**ROI 里“投入”到底包括什么？**

不同团队口径差很大，必须问清楚/主动说明：

1. **纯广告ROI**：

* 投入=广告消耗（巨量千川消耗）

2. **全成本ROI（更适合资金安全/补贴风控）**：

* 投入=广告消耗 + 补贴/券成本 + 返利成本 +（可能还有）达人佣金/服务费

3. **结算ROI**：

* 产出用净GMV或结算GMV，更防“先冲量后退款”的虚高ROI

**典型“口径坑”（面试最爱考）**

**坑1：用下单GMV算ROI会被“退款/羊毛”放大**

* 下单GMV很漂亮，但后面大量退款 → ROI 虚高
  ✅ 风控更偏向：**支付GMV/净GMV/结算GMV**做护栏

**坑2：只看ROI，忽略规模**

* ROI 很高但 GMV 很小，可能是过严策略或样本偏差
  ✅ 通常要一起看：**GMV（规模）+ ROI（效率）**

**坑3：归因窗口不同导致ROI不可比**

* 7天归因 vs 1天归因，ROI能差很多
  ✅ 对齐：**归因窗口、去重规则、跨端归因、自然流量是否剔除**

**在“风控策略运营”里怎么用 GMV/ROI？**

你可以用这套表达：

* **GMV 是增长目标，ROI 是效率目标，风控的目标是在控制风险损失的前提下尽量不伤 GMV/ROI。**
* 策略评估要“三看”：

  1. **挽损**（拦截/减少的作弊补贴、异常退款等）
  2. **误伤**（正常商家/用户被拦截导致 GMV/ROI 下滑）
  3. **净收益**：`净收益 = 挽损 - 误伤损失（GMV/利润影响） - 运营/审核成本`



**5）遇到“对抗升级”（策略刚上线就失效），你怎么迭代？**

**答题结构：**

* 快速止血：临时规则/限额/验证码/提高成本
* 分析对抗路径：命中样本复盘（新增绕过点）
* 提升泛化：用组合特征、序列特征、关系特征，减少可被单点规避的规则
* 工程化：黑样回流、周迭代节奏、告警驱动的自动回归测试

## SQL / 数仓题

**1）SQL：找出最近7天“补贴消耗异常”的商家（高于历史均值+3σ）**

**示例思路：**

* 先算商家在“历史窗口”的均值和标准差
* 再算最近7天的消耗
* 对比阈值输出 Top 商家

```sql
WITH hist AS (
  SELECT
    merchant_id,
    AVG(subsidy_cost) AS mu,
    STDDEV_POP(subsidy_cost) AS sigma
  FROM dwd_merchant_day
  WHERE dt BETWEEN DATE_SUB(CURRENT_DATE, 60) AND DATE_SUB(CURRENT_DATE, 8)
  GROUP BY merchant_id
),
recent AS (
  SELECT
    merchant_id,
    SUM(subsidy_cost) AS cost_7d
  FROM dwd_merchant_day
  WHERE dt BETWEEN DATE_SUB(CURRENT_DATE, 7) AND CURRENT_DATE
  GROUP BY merchant_id
)
SELECT
  r.merchant_id, r.cost_7d, h.mu, h.sigma
FROM recent r
JOIN hist h ON r.merchant_id = h.merchant_id
WHERE r.cost_7d > (h.mu * 7 + 3 * h.sigma * SQRT(7))
ORDER BY r.cost_7d DESC;
```

**2）SQL：计算每个投放计划的漏斗（曝光→点击→下单→支付），并输出转化率**

```sql
SELECT
  plan_id,
  COUNT(DISTINCT CASE WHEN event='impression' THEN user_id END) AS uv_imp,
  COUNT(DISTINCT CASE WHEN event='click' THEN user_id END)      AS uv_clk,
  COUNT(DISTINCT CASE WHEN event='order' THEN user_id END)      AS uv_ord,
  COUNT(DISTINCT CASE WHEN event='pay' THEN user_id END)        AS uv_pay,
  1.0 * COUNT(DISTINCT CASE WHEN event='click' THEN user_id END)
      / NULLIF(COUNT(DISTINCT CASE WHEN event='impression' THEN user_id END),0) AS ctr,
  1.0 * COUNT(DISTINCT CASE WHEN event='pay' THEN user_id END)
      / NULLIF(COUNT(DISTINCT CASE WHEN event='click' THEN user_id END),0) AS cvr_click2pay
FROM dwd_plan_event
WHERE dt = CURRENT_DATE
GROUP BY plan_id;
```


## Python/分析题

**1）给你一堆订单/补贴数据，如何用 Python 快速做“异常点定位”？**

**回答框架：**

1. 数据清洗：去重、口径统一、异常值处理
2. 分层聚合：按天/小时、类目、商家、计划聚合
3. 异常检测：z-score、IQR、EWMA、季节性分解（有周期就用STL）
4. 贡献度分析：定位Top贡献（比如商家贡献了60%增量）
5. 输出可视化与结论：看板 + 结论 + 建议策略

**2）如何做“实时异常波动”而不是事后统计？**

**关键点：**

* 用流式聚合（分钟级/5分钟级），维护基线（滑动均值、分位数）
* 告警要有“抑制”：同一实体短时间不要重复打爆
* 产物：实时看板 + 告警推送 + 一键下钻（商家/计划/地域）

## 实验与上线题

**1）策略上线怎么做灰度？怎么判断“可全量”？**

**标准答案：**

* 灰度维度：按流量比例、类目、商家分层（优先低风险/可控场景）
* 观察期：至少覆盖业务周期（工作日/周末/大促前后）
* 判定：损失下降达标、误杀/申诉可控、GMV/ROI 不劣化（或在可接受范围）
* 有回滚：开关、阈值可热更新、兜底策略

**2）如果业务方说“你们风控太严影响增长”，你怎么沟通？**

**建议答法：**

* 先对齐目标：增长与资金安全都重要
* 用数据说话：给出误杀率、GMV影响、挽损金额，展示“收益-成本”
* 提供选项：不同阈值的 trade-off 曲线，让业务选择可接受点
* 提供保护措施：白名单、申诉快通道、分层策略（优质商家更宽）
