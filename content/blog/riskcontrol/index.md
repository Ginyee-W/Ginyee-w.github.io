## 业务风控题

**1）如何定义“营销资金安全”的核心目标？你会看哪些指标？**

**答题要点：**

* 目标不是“零风险”，而是**在可控风险下最大化业务收益**（GMV/ROI/投放规模）。
* 指标分三层：

  * **损失类**：疑似/确认作弊金额、赔付金额、异常退款金额、补贴/返利被套利金额
  * **业务影响类**：误杀率（拦截正常商家/达人/用户）、放行风险率、转化率/GMV/ROI 变化
  * **效率类**：命中率、处置时延（发现→拦截→复盘）、人工审核量/通过率

**一句话总结**：用“损失-收益-体验-效率”四象限建立北极星 + 护栏指标。

**2）发现某类产品在推广投放后的退款率、补贴消耗突然飙升，你怎么排查？**

**思路框架（面试很加分）：**

1. **确认口径**：退款率分母是下单还是支付？时间窗是下单日还是退款日？
2. **分层定位**：按类目→商家→计划→素材→人群包→地域/设备→渠道/联盟分解贡献（Top N）
3. **对照组**：和历史同周期、同类目、同活动（大促/补贴政策变化）对比
4. **判断性质**：

   * 活动/策略变更导致的“真波动”
   * 交易链路异常（支付、券核销）
   * 风险行为集中爆发（异常账号/订单模式）
5. **处置方案**：先止血（限额/灰度/提高门槛）→再定性（规则/模型/审核）→复盘沉淀指标

**3）“风险与业务收益平衡”怎么做？给一个可落地的方法**

**可落地做法：**

* 把策略看成一个二分类拦截器：阈值越严 → 损失下降但误杀上升、GMV 下降。
* 选择阈值的方法：

  * 设定**成本矩阵**：放过 1 单风险损失 = L；误杀 1 单正常损失 = B（GMV/ROI/商家体验）
  * 目标：最小化 `L*FN + B*FP` 或最大化 `收益 - 风险损失 - 误杀损失`
* 实操：**灰度+A/B**，在不同阈值下对比“拦截金额、误杀率、GMV/ROI”。


**4）如果你要做“黄牛/羊毛党”治理，策略体系怎么搭？**

**建议答案结构：**

* **分阶段**：准入（注册/实名/设备）→行为（浏览/加购/下单/支付/退款）→履约（签收/退货）→资金（补贴/返利/提现）
* **分手段**：规则（强约束）+ 模型（软识别）+ 机制（成本提高）+ 联防（黑名单/风控中台/商家侧）
* **分人群**：新客/老客、达人/商家、计划/素材，分别设置不同阈值和运营策略（例如新客更谨慎但需要留出增长空间）


**5）你会如何写一份“防控方案需求文档”（PRD/BRD）？**

**高频考点**：跨团队协作与项目推进能力
**模板回答：**

* 背景与问题：风险现象、规模、影响范围
* 目标与指标：损失下降 X%、误杀 < Y%、审核量 < Z
* 方案：策略链路（命中→处置→申诉→复核）、规则/模型/名单、阈值与灰度
* 数据需求：字段、埋点、口径、看板
* 风险与兜底：误杀申诉、回滚机制、紧急开关
* 排期与分工：研发/数仓/算法/运营/法务（如需）



**6）“实时监控异常波动”你会怎么做报警体系？**

**要点：**

* 指标分层：全站 → 类目/业务线 → 商家/计划 → 关键链路
* 报警规则：同比/环比、分位数阈值、EWMA/异常检测、突变点
* 报警质量：降噪（合并告警、抑制、冷却时间）、可解释（贡献度 Top N）
* 处置闭环：告警→工单→结论（真异常/误报）→规则迭代

**7）如果你是灰产或者刷单的商家，你会怎么反对抗风控系统”**

**1.对抗的“方向”通常是什么（高层次抽象）**

> 攻击者通常会尝试：

* **伪装成正常行为**：让行为分布更像真实用户/真实商家
* **分散与低频**：把风险信号稀释到多账号、多计划、多链路
* **团伙协同**：利用账号-设备-地址-支付工具的关联做“分工”
* **利用规则/政策缝隙**：比如活动规则、补贴门槛、结算/退款时间窗
* **试探系统边界**：小流量测试阈值，找到“刚好不触发”的区域

**2.我们怎么防（分层体系，而不是单点规则）**

> 我会把防护做成“多层+多信号”的体系：

* **准入与门槛层**：资质/行为门槛、分层额度、冷启动更谨慎
* **实时拦截层**：规则底线 + 风险评分排序，命中后分级处置（限额/二次校验/延迟结算等）
* **关系与团伙层**：做实体关联图谱（账号-设备-地址-支付等），抓“组织化风险”
* **事后追溯层**：结算/退款后验，黑样回流，形成样本闭环
* **动态对抗层**：灰度发布、阈值动态化、告警驱动迭代，避免策略“一刀切、被摸透”

**3.怎么证明有效（量化验收）**

> 用 A/B 或灰度看三类指标：

* **挽损**：异常补贴/退款/作弊金额下降
* **体验护栏**：误杀率、申诉通过率、GMV/ROI 影响
* **运营效率**：人工审核量、处置时延、告警质量

本质上风控**不是追求“永远抓完”**，而是通过对抗思维把**灰产成本抬高**、把**发现速度变快**，同时把对正常业务的影响控制在可接受范围。

**8）先用后付违约风险控制**

“先用后付”本质是一个**小额短周期的信用支付**场景，违约风险控制可以按“**准入 → 额度 → 交易风控 → 事后催收 → 复盘迭代**”来搭体系。下面给你一套面试/方案讨论时很通用、也足够落地的框架。

**1. 风险类型拆解（先把问题讲清楚）**

先用后付的“违约”不止一种：

* **真实违约**：用户确实没钱/不想还（现金流问题、主观赖账）
* **欺诈违约**：羊毛党/黑产批量注册套取“先乘后付”额度
* **争议类违约**：对费用/路线/服务不满，拒付或拖延（非纯信用风险）
* **账户被盗/代付纠纷**：账号被盗刷、他人乘车
* **支付链路失败**：扣款失败、银行卡/支付账户异常（可通过重试/换绑解决）

> 面试里这样说很加分：**先分类，后对每类设计不同策略**，否则会把“催收”和“反欺诈”混在一起。

**2.准入策略（谁能开通“先用后付”）**

目标：把高风险人群挡在门外，同时不伤正常转化。

常见做法（不涉及可被滥用的细节）：

* **分层准入**：老用户优先（有稳定完单&支付历史）；新用户先走更保守策略
* **实名/绑定强度**：实名、稳定支付工具绑定、设备稳定性等作为基础门槛
* **黑名单/灰名单**：历史欠款、异常申诉、团伙关联实体命中直接拒绝或限额
* **冷启动保护**：新开通用户给很小额度 + 强监控，逐步放开

**3.额度与敞口管理（核心：别让损失变大）**

先用后付最关键是“**控敞口**”，不是指望 100% 不违约。

* **授信额度**：按用户风险等级给额度（如：可欠费上限、可未结订单数）
* **动态额度**：随着行为更新（连续按时还款→提额；多次扣款失败→降额/冻结）
* **单笔/单日/单周限额**：尤其对长途/高价订单更敏感
* **风险保守项**：遇到高风险信号时触发“先付/部分预付/保证金/二次验证”等“升阶动作”

> 面试一句话：**“风险=PD×LGD×EAD，我主要用EAD（敞口）快速把损失封顶。”**

**4. 交易风控（下单/行程中/计费完成三个时点）**

把风控做成**多时点决策**，而不是只在开通时看一次。

**A. 下单前（Pre-check）**

* 实时风险评分：用户画像 + 行为/设备/地理位置稳定性 + 历史支付表现
* 高风险订单触发升阶：改为先付/部分预付/限制使用先用后付

**B. 行程中（In-trip）**

* 异常检测：路线/时长/价格突增、频繁改派/取消、异常高频下单
* 风险升阶：必要时提示改为先付或限制后续下单（更偏“止损”）

**C. 计费后（Post-trip）**

* 自动扣款策略：多通道尝试、合理重试、提示换绑/补足
* 争议处理：把“投诉争议”与“恶意拒付”分流，避免误伤好用户

**5.事后催收与处置（效率 + 体验平衡）**

先用后付金额通常不大，催收要“轻量化、自动化”：

* **提醒链路**：站内信/短信/Push → 引导一键补缴/换绑
* **自动扣款重试**：合理频率 + 失败原因归因（余额不足/账户异常）
* **使用限制**：未结清前限制使用先用后付、限制下单频率/某些高价场景
* **升级处置**：长期欠款进入更严格处置（注意合规与用户协议边界）

**6. 反黑产：团伙与批量化识别（防“薅额度”）**

这块通常是风控的“收益大头”：

* **实体关联图谱**：账号-设备-支付工具-手机号-地址-常用上车点等做关联
* **团伙特征**：短期批量注册、模式高度一致、共享资源（设备/支付/位置）
* **规则 + 模型组合**：规则挡底线，模型做排序与泛化，人工复核兜底

**7. 评估指标（面试一定要讲清楚怎么验收）**

至少三类指标：

1. **风险指标**

* 逾期率（DPD1/3/7/30）、坏账率、挽回率、单均损失

2. **业务护栏**

* 开通率、使用率、完单率、GMV、转化、用户投诉率/申诉通过率（衡量误伤）

3. **运营效率**

* 人工介入率、处置时延、告警准确率

> 很加分的表达：**“用灰度/A-B 验证：风险下降的同时，GMV/完单率不能显著劣化。”**

## 建模题

**1）规则和模型怎么选？什么时候“规则优先”，什么时候“模型优先”？**

**推荐回答：**

* 规则优先：风险形态明确、强约束、需要可解释/可审计、冷启动
* 模型优先：风险形态多变、规则难覆盖、需要更高召回/自动化
* 最佳实践：**规则做底线（硬拦截/限额）+ 模型做排序（风险评分）+ 人审做兜底**



**2）如果让你做一个“风险评分卡”，你会用哪些特征？**

**按实体分组说更像专业风控：**

* 账号/主体：注册时长、实名状态、历史违规、关联账号数
* 设备/环境：设备指纹稳定性、IP/地域漂移、代理/异常网络信号
* 行为序列：短时间高频操作、路径异常（跳步）、夜间集中
* 交易/资金：客单价分布异常、退款率、补贴占比、同地址/同设备多单
* 关系网络：账号-设备-地址-支付工具的关联图谱（团伙识别）

（面试时强调：**特征要可落库、可实时、可解释**）



**3）怎么评估一个风控策略/模型？只说AUC够吗？**

**答：不够。**

* 风控更关心：Precision/Recall、KS、PR-AUC、拦截金额/挽损、误杀率、申诉通过率、人工审核节省
* 线上还要看：GMV/ROI 影响、用户/商家体验指标（下单转化、支付成功率）
* 强调：**离线指标 + 线上业务指标**一起验收
GMV 和 ROI 在电商投放/风控里几乎是天天挂在嘴边的两个核心指标，我把**定义、公式、口径坑点、以及风控场景怎么用**一次讲清楚。

**4）GMV/ROI 是什么？**

**GMV（Gross Merchandise Value）= 成交总额 / 交易总额**
通常指一段时间内产生的订单金额总和（偏“规模”）。

常见口径

* **下单GMV**：按“下单时间”统计订单金额（含未支付/后续退款的单，视平台口径）
* **支付GMV**：按“支付时间”统计实际支付金额
* **确认收货GMV**：按“确认收货/结算”口径，更贴近最终收入
* **净GMV（Net GMV）**：扣除退款/取消后的 GMV

> 面试时可以主动说一句：**“我会先对齐GMV口径：下单/支付/净额/结算”**，很加分。

**ROI（Return on Investment）= 投入产出比**

在投放里一般理解为：**花出去的广告费/补贴，带回来的成交金额**（或收益）。

**最常见（投放）ROI 口径**

* **ROI = GMV / 广告消耗（Spend）**

  * 例：Spend=10万，GMV=50万 → ROI=5

**也会出现的变体（更接近利润）**

* **利润ROI = 利润 / 投入**
* **边际ROI（mROI）**：增加 1 块钱投放，GMV/利润多了多少（用于调控）

**ROI 里“投入”到底包括什么？**

不同团队口径差很大，必须问清楚/主动说明：

1. **纯广告ROI**：

* 投入=广告消耗（巨量千川消耗）

2. **全成本ROI（更适合资金安全/补贴风控）**：

* 投入=广告消耗 + 补贴/券成本 + 返利成本 +（可能还有）达人佣金/服务费

3. **结算ROI**：

* 产出用净GMV或结算GMV，更防“先冲量后退款”的虚高ROI

**典型“口径坑”（面试最爱考）**

**坑1：用下单GMV算ROI会被“退款/羊毛”放大**

* 下单GMV很漂亮，但后面大量退款 → ROI 虚高
  ✅ 风控更偏向：**支付GMV/净GMV/结算GMV**做护栏

**坑2：只看ROI，忽略规模**

* ROI 很高但 GMV 很小，可能是过严策略或样本偏差
  ✅ 通常要一起看：**GMV（规模）+ ROI（效率）**

**坑3：归因窗口不同导致ROI不可比**

* 7天归因 vs 1天归因，ROI能差很多
  ✅ 对齐：**归因窗口、去重规则、跨端归因、自然流量是否剔除**

**在“风控策略运营”里怎么用 GMV/ROI？**

你可以用这套表达：

* **GMV 是增长目标，ROI 是效率目标，风控的目标是在控制风险损失的前提下尽量不伤 GMV/ROI。**
* 策略评估要“三看”：

  1. **挽损**（拦截/减少的作弊补贴、异常退款等）
  2. **误伤**（正常商家/用户被拦截导致 GMV/ROI 下滑）
  3. **净收益**：`净收益 = 挽损 - 误伤损失（GMV/利润影响） - 运营/审核成本`

**刷单情景下的数据和指标特征选择**

**刷单风控的特征选取：按“实体 + 关系 + 时间”三维来分**

* **A. 买家（User）侧特征**

* **账号质量**：注册时长、实名/绑定强度、历史下单/支付成功率、历史退款/拒付
* **行为路径**：曝光→点击→加购→下单→支付的耗时分布（异常“过快/过一致”）、夜间占比
* **购买结构**：新客占比异常（大量“首单”集中在某商家/某SKU）、跨店购买多样性低
* **复用特征**：同一买家短窗多单、多店同款、订单间隔极短（velocity）

* **B. 商家（Merchant）侧特征**

* **成交爆发**：分钟/小时级订单突增、波动远超自身历史基线
* **用户结构异常**：新客比例、低活跃用户比例、同地域/同设备来源占比异常
* **商品结构**：少数SKU贡献过高（Top1/Top3成交占比）、低价SKU异常拉量
* **售后结构**：退款率/取消率/仅退款占比、售后发生时延分布异常

* **C. 订单（Order）侧特征**

* **金额/补贴结构**：券/补贴占比、实付占比异常、固定价位密集（金额“扎堆”）
* **支付成功模式**：支付失败→重试→成功的模式是否异常一致
* **地域一致性**：收货地/定位/常驻地匹配度（只做风控信号，不做个人敏感推断）

* **D. 设备/网络（Device & Network）侧特征（反批量最关键）**

* **设备指纹稳定性**：同设备关联多账号/多买家
* **IP/网络形态**：IP复用度、地域漂移、异常网络环境信号
* **多实体共享资源**：同设备/同IP/同支付工具关联到多买家/多订单/多商家（强团伙信号）

* **E. 履约/物流（Fulfillment）侧特征（电商特别有效）**

* **地址复用**：同地址对应多账号、地址多样性低
* **签收/退货模式**：签收时延分布异常、退货集中爆发、空包/异常物流状态（如果平台可得）

* **F. 关系网络（Graph）特征（抓“组织化刷单”）** 把 “买家-设备-地址-支付工具-商家-SKU” 做成图：

* **共享节点数**：一个设备/地址连接的账号数
* **二部图指标**：买家-商家图的密集团（bipartite density）、共同邻居数
* **团伙特征**：同一小团体在短时间对多个商家“复制行为模式”

**指标构建（Feature Engineering）**

* **A. 基线偏离类（对比历史/同类目）**

* **订单爆发度**：`(当前小时订单 - 历史均值) / 历史标准差`
* **新客占比偏离**：当前新客占比 vs 过去N天/同类目分位数
* **退款率偏离**：分“下单口径/支付口径/净额口径”避免被口径坑

* **B. 集中度/多样性类（刷单常“集中且整齐”）**

* **TopK贡献度**：Top1/Top3 SKU 的 GMV/单量占比
* **熵/多样性**（很好用也好讲）

  * 地址熵、设备熵、地域熵：越低越可疑（来源太集中）
* **Gini/HHI**：衡量订单来源是否过度集中

* **C. 速度/序列类（短时间批量化）**

* **Velocity**：同买家/同设备/同地址在 5min/1h 的下单数、支付数
* **时间间隔分布**：下单→支付间隔是否异常“过短且方差极小”
* **重复模式**：同路径点击序列重复度高（可用相似度/模板匹配）

## Python/分析题

**1）给你一堆订单/补贴数据，如何用 Python 快速做“异常点定位”？**

**回答框架：**

1. 数据清洗：去重、口径统一、异常值处理
2. 分层聚合：按天/小时、类目、商家、计划聚合
3. 异常检测：z-score、IQR、EWMA、季节性分解（有周期就用STL）
4. 贡献度分析：定位Top贡献（比如商家贡献了60%增量）
5. 输出可视化与结论：看板 + 结论 + 建议策略

**2）如何做“实时异常波动”而不是事后统计？**

**关键点：**

* 用流式聚合（分钟级/5分钟级），维护基线（滑动均值、分位数）
* 告警要有“抑制”：同一实体短时间不要重复打爆
* 产物：实时看板 + 告警推送 + 一键下钻（商家/计划/地域）

## 实验与上线题

**1）策略上线怎么做灰度？怎么判断“可全量”？**

**标准答案：**

* 灰度维度：按流量比例、类目、商家分层（优先低风险/可控场景）
* 观察期：至少覆盖业务周期（工作日/周末/大促前后）
* 判定：损失下降达标、误杀/申诉可控、GMV/ROI 不劣化（或在可接受范围）
* 有回滚：开关、阈值可热更新、兜底策略

**2）如果业务方说“你们风控太严影响增长”，你怎么沟通？**

**建议答法：**

* 先对齐目标：增长与资金安全都重要
* 用数据说话：给出误杀率、GMV影响、挽损金额，展示“收益-成本”
* 提供选项：不同阈值的 trade-off 曲线，让业务选择可接受点
* 提供保护措施：白名单、申诉快通道、分层策略（优质商家更宽）

**3）遇到“对抗升级”（策略刚上线就失效），你怎么迭代？**

**答题结构：**

* 快速止血：临时规则/限额/验证码/提高成本
* 分析对抗路径：命中样本复盘（新增绕过点）
* 提升泛化：用组合特征、序列特征、关系特征，减少可被单点规避的规则
* 工程化：黑样回流、周迭代节奏、告警驱动的自动回归测试

面试被问“**如何利用 AI agent 做特征工程/指标筛选/评估**”，最稳的答法是把它讲成一个**可上线的工作流**：Agent 不负责“拍脑袋决定”，而是负责**自动化探索、计算、验证、产出报告**，人来做最终决策 + 灰度。

下面给你一套可以直接背的结构（偏风控/营销场景也通用）。

**4）AI agent 的使用**

你可以先抛一句：

> 我会把 AI agent 当成“可控的自动化分析与实验编排器”：负责生成候选特征、跑统计与实验、做泄露/稳定性检查、输出可解释报告；最终上线阈值和策略仍由人审 + 灰度验收决定。

这句很加分：体现合规、可控、工程意识。

**Step A：输入目标与约束（Agent 的“任务边界”）**

* 目标：提升挽损/降低逾期/降低刷单等（明确 KPI）
* 护栏：误杀率、GMV/ROI、投诉率、审核量等
* 约束：实时性（T+0 / T+1）、字段可用性、合规范围

**Agent 做的事**：把这些变成一份“实验计划 + 指标定义（metric spec）”。

**Step B：候选特征生成（feature ideation），但用“模板化”避免胡编**

Agent 不直接瞎想，而是按 **实体×时间窗×聚合×偏离基线** 的模板生成：

* 实体：用户/商家/订单/设备/地域/商品/关系图谱
* 时间窗：5min/1h/1d/7d
* 聚合：count/sum/ratio/entropy/topk share/z-score
* 基线：对比历史均值、同类目分位数、同人群分层

**产物**：候选特征清单（带口径、SQL 伪代码、计算成本预估）。

**Step C：自动计算与画像（profiling）——让 agent 跑“第一轮体检”**

对每个候选特征，Agent 自动跑：

* 覆盖率/缺失率、分布（分位数、长尾）、异常值比例
* 与标签/损失的单变量关系（AUC/KS/IV、PR-AUC、Lift@TopK）
* 分层效果（新客/老客、类目、地域、设备）看是否“只在某个切片有效”

**关键点**：Agent 还要输出“为什么可能有用”的解释（便于 reason code）。

**Step D：自动化“风控必做三检”：泄露、稳定、冗余**

这一步是面试官最想听的“工程严谨性”。

1. **泄露检查（Leakage）**

* 校验特征的时间戳：必须在决策点之前可得
* 排除明显后验字段（比如退款最终状态去预测事前风险）
* 时间切分验证（按周/月）避免“离线虚高”

2. **稳定性/漂移（Stability & Drift）**

* PSI/分布漂移、不同周的性能波动
* 大促/活动前后是否崩掉（场景鲁棒性）

3. **冗余与共线性（Redundancy）**

* 相关性聚类 / VIF / permutation importance
* 留更稳定、更便宜、更可解释的那个

**Agent 的价值**：把这些检查自动化、标准化，减少漏检。

**Step E：多特征筛选与增量评估（Ablation + 线上灰度计划）**

Agent 组织实验而不是只报离线分：

* 基线模型/规则 → 加入一组特征 → 看 **Δ挽损、Δ误杀、Δ审核量**
* Ablation：逐个移除看边际贡献（marginal gain）
* 产出“候选特征 Top N + 推荐阈值范围 + 风险提示（可能误伤的人群）”

然后给出线上灰度方案：

* 灰度维度（类目/商家/流量比例）
* 观察期（至少覆盖工作日+周末/活动）
* 通过标准（挽损达标 + 护栏不劣化 + 申诉可控）

**Step F：产出可复用资产（这才叫“可上线”）**

Agent 最终输出三份东西：

1. **Feature spec 文档**：口径、窗口、SQL、依赖、延迟、监控项
2. **自动化单元测试**：空值、范围、分布漂移阈值、数据延迟告警
3. **看板与告警**：特征覆盖率/PSI、模型命中率、误杀/申诉趋势

**你可以提一个“多 Agent 分工”版本（更像真正落地）**

* **Profiler Agent**：数据质量/分布体检
* **Feature Builder Agent**：按模板生成并落 SQL/特征定义
* **Validator Agent**：泄露/稳定/合规检查 + 单元测试生成
* **Evaluator Agent**：离线评估 + ablation + 分层分析
* **Reporter Agent**：输出 PRD/实验报告/上线建议 + 原因码总结

人类负责：目标权衡、阈值拍板、灰度策略、申诉机制与复盘。
